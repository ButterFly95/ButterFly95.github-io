Web 应用黑客手册/第一章：Web 应用安全

Web 应用安全

Web 应用的发展

在互联网早期，只有 Web 网站，这些网站存放了一些静态文件。浏览器用于获取和显示文件。信息只从服务器流向浏览器。大部分网站不需要鉴别用户，因为不需要这么做，平等对待所有用户，展示相同的信息。当时的安全漏洞主要都是 Web 服务器软件的漏洞。如果骇客入侵了 Web 服务器，几乎得不到任何隐私信息，因为网站上的信息都是公开的。

时至今日，互联网已然变了模样。大部分网站都是一个 Web 应用。功能繁多，且信息在浏览器和服务器之间双向流动。这些网站支持注册，登录，交易，搜索，创作等。向用户呈现的内容是动态生成的，并且通常是针对每个用户定制的。（千人千面，个性化推荐。——本人注）很多信息都是隐私。于是乎，安全就成了一个问题。（中国人多数情况下愿意用隐私换便利——李彦宏）

每个 Web 应用都是不同的，可能包含独有的安全漏洞。为了提供核心功能，Web 应用通常需要处理大量用户隐私。如果一个骇客侵入了 Web 应用，他可以偷窃用户隐私，甚至用户财产，还可以攻击其他用户。

Web 应用公共功能

Web 应用几乎可以让人在线做任何线下可以做的事。如今主要的 Web 应用如下：

  • 购物
  • 社交
  • 银行
  • 搜索
  • 转账
  • 赌博
  • 博客
  • 邮箱
  • 信息（百科）

大部分 Web 应用客户端，包括手机 APP，网页，自定义客户端等，几乎都使用基于 HTML 的 API 和服务器进行通信。Web 应用的数据和 API 在不同客户端之间共享。

除了公网，Web 应用也广泛应用在各个公司的内网。包括：

  • HR 应用
  • 关键设置管理接口
  • 协作软件
  • 商业应用，如 ERP
  • 很多之前的软件支持浏览器访问，比如 Office

在以上的例子中，很多被认为是内部服务的应用正在向外部迁移以降低成本。在所谓的云解决方案中，商业应用的关键功能和数据比之前更加公开，攻击面扩大了。

大多数用户需要的唯一客户端软件就是浏览器的时代就要到了（JS在吞噬一切？）。各种不同的应用都依赖于一组共享的协议和技术，从而产生一组公共安全漏洞。

Web 应用的好处

几种技术因素和商业动机共同推动了互联网的革命：

  • HTTP
  • 每个用户的电脑和手机上都有浏览器
  • 如今的浏览器功能强大，可以构建丰富的用户界面
  • 用于开发 Web 应用程序的核心技术和语言足够简单

Web 应用安全

“网站是安全的”

作者通过数据说明，网站不安全。

安全核心问题：用户可以提交任意输入

因为客户端不受控制，因此，用户可以提交任意输入到 Web 应用的服务端。应用必须假设用户的输入是恶意的，从而必须采取措施确保骇客不能使用精心构造的数据来干扰应用程序的逻辑和行为，从而获取会应用的未授权访问。

这一核心问题体现在以下几个方面：

  1. 用户可以干扰客户端和服务端信息传输中的任意部分，包括，参数，HTTP头，cookies 等。在客户端采取的任何安全措施都可以轻松规避。
  2. 用户可以按照任意顺序发送请求，并且可以在应用程序的任意阶段提交参数，或者根本不提交。程序员对用户交互行为的任意假设都可能是错的。
  3. 用户不仅可以使用浏览器访问应用。还有很多其他工具可以和应用进行快速交互，产生大量请求来寻找漏洞。

大多数针对 Web 应用的攻击都涉及将精心设计的输入发送到服务器，从而导致应用出现设计者所未预料的行为。有一些例子如下：

  • 改变产品价格从而低价购买商品
  • 修改会话令牌来劫持其他经过认证的用户的会话
  • 删除特定参数，导致应用出现逻辑缺陷
  • 更改由后端数据库处理的输入，插入恶意数据查询访问隐私数据

新的安全边界

传统的基于 IP 的安全边界不适用了，因为 Web 服务是面对公开的。攻击者并不需要控制基础设施，只需要影响应用执行就足够了。所以，从服务端来看，安全从网络边界向后移动了，要保证 Web 应用的安全，就得保证 Web 应用的代码质量。

从另一方面看，当用户访问有缺陷的应用时，客户端安全也面临威胁。骇客可以控制服务端向客户端发送伪造信息。

Web 安全的未来

SQL 注入漏洞慢慢销声匿迹，新的更加微妙的漏洞依然可以发现。

攻击目标从服务端转向客户端。

总结

Web 安全的两个根本因素：

  • Web 应用的功能越来越多，领域越来越广，应用越来越复杂
  • 用户输入不受限制

深入理解 async

深入理解 async

本篇文章主要来源于 https://docs.microsoft.com/en-us/dotnet/standard/async-in-depth

在 C# 中进行异步编程是相对容易的，主要通过 Task 和 Task<T> 类型, async 和 await 关键字来实现。

Task 和 Task<T>

Task 代表即将会做的任务，Task<T> 表示该任务返回值的类型是 T，如果只写 Task 就代表该任务无返回值，相当于 Task<void>。

需要注意的是，Task 是对异步执行的任务的抽象，而不是对线程的抽象。默认情况下，Task 在当前线程执行，在必要的时候，Task 会在别的线程执行。可以通过Task.Run API 来强制一个 Task 在别的线程执行，但是，这并不代表会开启新线程。

I/O 密集型应用

一次 IO 的完整过程如下：

  1. 线程 A 调用操作系统接口，驱动代码把 IO 请求放在 IO 请求队列上，此时当前线程为 A
  2. 硬件设备处理 IO 请求，此时线程不一定是 A
  3. 硬件设备处理完 IO 请求后产生中断信号，执行中断处理程序，IO 请求的数据交给系统，此时的线程也不一定是 A
  4. 线程 A 成为当前线程，系统执行 IO 回调函数，把 IO 请求的返回数据给线程 A

其中，步骤 2 占据整个过程 90% 以上的时间。

操作系统提供 2 种 IO 接口：同步和异步。二者的主要区别在于：

  1. 使用同步 IO 的线程在执行完步骤 1 之后，会被阻塞，之后会切换线程；使用异步 IO 的线程在执行完步骤 1 之后，不会被阻塞
  2. 使用异步 IO 需要注册回调函数，在步骤 4，系统会执行该回调函数，回调函数入参包括 IO 请求返回的数据；使用同步 IO 不需要注册回调函数

对于 IO 密集型应用，使用同步 IO 必然是不可取的，因为

  1. 假如线程数量少，大部分时间线程都会被阻塞在步骤 2 和 3，浪费 IO 设备性能
  2. 假如线程数量多，就会出现大量的线程切换，浪费 CPU 性能

因此，对于 IO 密集型应用只能考虑异步 IO，传统的异步 IO 需要使用回调函数，编程体验很差，此处不多解释
