Web 应用黑客手册/第十章：攻击后端组件

攻击后端组件

系统命令注入

一般出现在定制的 Web 应用中，比如防火墙设备，这种设备一般需要执行直接和操作系统交互的命令

动态执行注入

很多脚本语言支持代码运行时动态执行。PHP 中的 exec 函数就可以动态执行代码。

寻找 OS 注入漏洞

在映射应用的时候，根据功能判断出需要进行系统交互的 URL。

有两类元字符：

  • ; | & 换行符 主要用于命令批处理
  • `

很多时候，命令的执行结果并不会直接返回，测试是否有漏洞的最好方法是使用时延命令。

Hack steps

  1. 换着花样使用 ping，测试是否存在命令注入漏洞
  2. 测试 ls 或者 dir 之类有返回结果的命令，看网页是否有返回
  3. 如果不能直接获取返回结果，可以考虑反向 shell，或者把返回结果写到文件再访问
  4. 确定用户的权限

如果有注入漏洞，但是命令被禁止，无法形成有效攻击，可以使用类似 nslookup 的命令

nslookup 查询失败会把域名打印出来，可以把脚本作为输入参数，把输出重定向到其他脚本文件，然后访问该文件。

如果不允许使用空格，在类 UNIX 系统中，可以使用 $IFS 代替。

寻找动态执行漏洞

动态执行漏洞多出现在 PHP 和 perl 程序中。

hack steps

  1. 任何用户提供的参数都可能是动态执行代码的一部分，尤其是服务器写的 cookie 参数。（我猜是过于信任 cookie 的安全）
  2. 输入 echo 命令
  3. 查看返回中是否有步骤 2 中 echo 输出的信息
  4. 查看是否有由于步骤 2 导致的错误信息，调整步骤 2
  5. 如果是 PHP 应用，可以考虑 phpinfo() 函数
  6. 如果确有漏洞，使用 system() 执行系统命令
防止系统命令注入

最好不要直接调用系统命令，实际上，应用中的任何功能都可以通过 API 实现。

如果不可避免，就考虑白名单，限制字符集。

执行命令使用命令 API 而不是传入字符串给 shell 执行。

防止脚本注入漏洞

最好不要把用户的输入传入动态执行函数中。不可避免的场景下，使用白名单。

操作文件路径

有很多功能都需要把用户输入的参数当作文件名或目录名。

状态是必须的

攻击认证

认证技术

  • HTML forms-based authentication
  • Multifactor mechanisms, such as those combining passwords and physical tokens
  • Client SSL certificates and/or smartcards
  • HTTP basic and digest authentication
  • Windows-integrated authentication using NTLM or Kerberos
  • Authentication services

90% 应用使用用户名和密码认证。

认证技术的设计缺陷

弱密码

许多 Web 应用不控制密码的质量

暴力登录

尝试常见密码

详细的错误信息

假如错误信息中告诉用户错误的是用户名或密码，可以借此收集用户名。

新用户注册，修改密码，也可以进行用户名枚举。

用户名和密码是多个输入参数的一种，假如在其他场景，比如多阶段功能，可以借助错误信息确认正确的参数。

有时候，返回错误的消息从浏览器看貌似一致，但是看 http 响应报文，则不一致。（能否根据报文处理时间来判断？）

除了枚举，还有其他方法来获取用户名，比如，邮箱，手机号，一般都是有效的用户名。

有漏洞的凭据传输
密码修改

如果匿名用户可以访问密码修改功能：

  • 可以用于猜测用户名
  • 如果不允许使用曾用密码，可以用于猜测密码
  • 验证完已有密码之后，验证新密码和确认密码是否一致
忘记密码

恢复密码问题比密码简单

记住我
  • 使用简单的持久化 cookie，rememberUser=xxx
  • 使用序列号cookie，rememberUser=111
  • 跨站脚本攻击劫持 cookie
凭据的不安全分发

比如，通过邮件发送密码或者激活链接，如果链接的构造可预测，就很危险

认证技术的实现缺陷

不正确的异常处理

提交各种异常输入来 fuzz 应用

多阶段登录机制的缺陷

多阶段登录的一般形式：

  1. 用户名 + 密码
  2. pin 码
  3. 物理设备产生的 code

多阶段登录容易引入的不安全假设：

  • 假设处于步骤三的用户必然已经完成步骤一和步骤二
  • 在步骤二阶段相信步骤一阶段服务器返回给用户的数据
  • 假设每个步骤的用户是同一个

可以在登录阶段，向用户提问曾经的问题，可能有的缺陷

  • 问题的答案也发送给客户端
  • 不记录用户没有回答出来的问题，用户可以反复尝试，执导遇到自己知道的问题
不安全的存储
  • 密码明文存储
  • 简单哈希存储

安全认证

保证凭据质量
  • 保证密码质量
  • 用户名唯一
  • 任何系统生成的数据都要有极高的信息熵，不可预测
  • 允许用户设置较长的密码
安全处理凭据
  • 在凭据的处理流程中，不能出现未授权访问
  • 客户端服务端通信使用 HTTPS
  • 通过 POST 请求发送凭据到服务端。凭据不能放在 URL 参数和 cookie 中。凭据不能回传给客户端。
  • 服务端不能存储明文密码
  • 客户端的记住我功能，只能存储用户名
  • 提供密码更改功能
  • 通过邮件发给用户的密码要有有效期
  • 使用小键盘输入密码
争取验证凭据
  • 验证流程的异常处理
  • 多阶段登录小心
  • 之前的验证结果不可信
防止信息泄露
  • 攻击者不能从错误信息中推测哪个参数错误
  • 用同一个错误信息标识各种类型的错误
  • 注册阶段用户名枚举：应用随机生成用户名或者邮箱注册

总结

我发现，密码的安全最终依赖手机和邮箱的安全啊。

Web 应用黑客手册/第五章：绕过客户端控制

绕过客户端控制

虽然 Web 安全的核心在于用户可以提交任意输入，但是，应用会在客户端做限制，突破限制才能提交任意输入。

通过客户端传输数据

这种场景很常见：服务端发送数据给客户端，之后，客户端又把该数据回传给服务端。

Why？

  • 减少服务端需要保存的会话数据，按照 REST 的原则，服务端不应该保存任何客户端的状态信息
  • 如果应用部署在不同的服务端，服务端需要的共享信息，需要借助客户端来维护
  • 如果应用部署了第三方组件，也需要借助客户端保存信息
  • 服务器保存信息的设计比较复杂，不利于测试，效率低，这也是 REST 受欢迎的原因吧

然而，通过这种方式传输敏感信息是很危险的。

隐藏表单属性

表单的隐藏属性虽然不能通过点击浏览器修改，但是，可以直接拦截并修改 HTTP 报文。

（猜测一下，假如某东的个性化售价没有在服务端做校验会怎样？）

Referer Header

多阶段功能，假如程序员对该字段作假设

不透明数据

猜测加密手段

捕获用户数据：HTML 表单

长度限制
脚本验证
禁止元素

以上都可以轻松突破

捕获浏览器插件数据

书中都是 java flash Silverlight 之类的东西，跳过

安全处理客户端数据

鉴于 REST 的设计，客户端必然要存储大量数据，隐私数据在客户端是不可避免的。

所以，服务端对客户端来的数据都要校验，对于那些绕过客户端安全校验的数据，都要记录。

Web 爬虫

Burp Suite

爬虫其实就是图遍历。麻烦之处在于有的网页只有在 post 之后才能看到。

robots.txt 是网站根目录下的文件，用于告诉爬虫工具，本站什么路径可以爬虫，什么路径不可以。在该文件中可以发现一下有趣的信息。比如，https://github.com/robots.txt，https://www.taobao.com/robots.txt

  • 不适应异常的导航机制，尤其是 js 比较多

  • 有的网站需要登录才能访问

潜在风险：有的网站比较垃圾，爬虫工具一个万一给删库了就得从入门到入狱了

手动爬虫

HTTP 请求

HTTP 请求报文一般包括以下几部分：

  1. HTTP 版本
  2. 状态码

  8. Content-Length，文档长度
HTTP Methods
  1. GET，获取资源
  2. POST，执行操作
  3. HEAD，等同没有响应消息没有 message body 的 GET
  4. TRACE，用于诊断
  5. OPTIONS，获取支持的方法
  6. PUT，上传文件

Web 应用黑客手册/第二章：核心防御机制

核心防御机制

处理用户访问

Web 应用需要满足的核心安全需要是，控制用户对应用数据和功能的访问。一般起码可以把用户分为三类：匿名用户，普通用户，管理员用户。除此之外，普通用户之间也有权限区别，比如，A 不能查看 B 的邮件。

大部分 Web 应用通过以下三种相关联的安全机制处理访问：

  • 身份认证
  • 会话管理
  • 访问控制

每个机制都是一个攻击面，且都是整体安全性的基础。因为它们的互相依赖性，总体的安全性取决于最薄弱的一个环节。任何一个环节的缺陷都可能导致骇客不受限制地访问应用。

身份认证

身份认证是应用处理用户访问最基础的依赖项。身份认证机制需要确认用户就是他所声称的那个人。如果无法鉴别用户，那么应用就只能把所有用户当作匿名用户，给他们最低级别的访问权限。

大部分 Web 应用都采用用户名加密码的机制来进行身份认证。在安全性要求更高的领域，还有其他机制来辅助，比如客户端证书，智能卡，挑战响应令牌等。除了核心登录功能，授权机制通常还包括其他辅助功能：自注册，账号恢复，密码修改等。

尽管认证机制看起来很简单，但是，通常会在涉及和实现中存在缺陷。常见的问题可能使得骇客可以识别其他用户用户名，猜测密码，绕过登录环节。

会话管理

用户登录成功后，就可以访问应用。为了进行访问控制，应用需要一种方式来区分不同的用户。

实际上，所有的 Web 应用都通过为每个用户创建一个会话，并为用户发布一个标识该会话的令牌来满足这一要求。会话是应用用于保存用户与应用当前交互状态的数据结构，令牌是用于映射该会话的唯一字符串。浏览器收到服务器发来的令牌后，会在之后的 HTTP 请求中附加该令牌。HTTP cookies 是令牌的标准实现。有的应用也会用隐藏表单或者 URL 参数来实现令牌。如果一个用户在一段时间内没有访问应用，那么令牌就会失效。（会话和令牌是逻辑层概念，cookie，表单或者 URL参数是实现的位置，具体的数据结构，都是字符串。——本人注）

在攻击面方面，会话管理高度依赖令牌的安全。针对会话管理的大多数攻击都试图窃取发给其他用户的令牌。如果成功的话，骇客就可以伪装成其他用户来访问应用。漏洞主要包括令牌生成缺陷，攻击者如果能猜测其他用户的令牌就可以伪装其他用户；令牌处理缺陷，攻击者可以窃取其他用户的令牌。

少数应用采用其他方式在多个 HTTP 交互中识别用户，不需要使用令牌。如果使用 HTTP 默认的身份认证机制，浏览器会在在每次请求时自动提交令牌。应用也可以把状态信息存储在客户端，通常都会加密。

访问控制

处理用户访问的最后一步就是判断用户的请求是否可以接受。在这一步，需要确定用户是否有权限执行某个操作，或者访问某些数据。

访问控制需要实现细粒度的逻辑。应用可能支持多种用户，每种用户的都有不同的权限。

因为典型访问控制的复杂性，这是安全漏洞的常见来源，攻击者可以获得对数据的未授权访问。开发人员经常对用户和应用的交互方式做出错误的假设，并且经常忽略某些功能的访问控制检查，从而导致未授权访问。探测这种漏洞通常很困难，因为基本上要对每个功能项重复相同的检查。然而，由于访问控制缺陷的普遍存在，这些工作通常是值得的。

处理用户输入

所有用户输入都是不可信的。大量安全漏洞都来自于对用户提交的意外输入。因此，Web 安全要求开发者用合适的方式处理用户输入。

Web 应用黑客手册/第一章：Web 应用安全

Web 应用安全

Web 应用的发展

在互联网早期，只有 Web 网站，这些网站存放了一些静态文件。浏览器用于获取和显示文件。信息只从服务器流向浏览器。大部分网站不需要鉴别用户，因为不需要这么做，平等对待所有用户，展示相同的信息。当时的安全漏洞主要都是 Web 服务器软件的漏洞。如果骇客入侵了 Web 服务器，几乎得不到任何隐私信息，因为网站上的信息都是公开的。

时至今日，互联网已然变了模样。大部分网站都是一个 Web 应用。功能繁多，且信息在浏览器和服务器之间双向流动。这些网站支持注册，登录，交易，搜索，创作等。向用户呈现的内容是动态生成的，并且通常是针对每个用户定制的。（千人千面，个性化推荐。——本人注）很多信息都是隐私。于是乎，安全就成了一个问题。（中国人多数情况下愿意用隐私换便利——李彦宏）

每个 Web 应用都是不同的，可能包含独有的安全漏洞。为了提供核心功能，Web 应用通常需要处理大量用户隐私。如果一个骇客侵入了 Web 应用，他可以偷窃用户隐私，甚至用户财产，还可以攻击其他用户。

Web 应用公共功能

Web 应用几乎可以让人在线做任何线下可以做的事。如今主要的 Web 应用如下：

  • 购物
  • 社交
  • 银行
  • 搜索
  • 转账
  • 赌博
  • 博客
  • 邮箱
  • 信息（百科）

大部分 Web 应用客户端，包括手机 APP，网页，自定义客户端等，几乎都使用基于 HTML 的 API 和服务器进行通信。Web 应用的数据和 API 在不同客户端之间共享。

除了公网，Web 应用也广泛应用在各个公司的内网。包括：

  • HR 应用
  • 关键设置管理接口
  • 协作软件
  • 商业应用，如 ERP
  • 很多之前的软件支持浏览器访问，比如 Office

在以上的例子中，很多被认为是内部服务的应用正在向外部迁移以降低成本。在所谓的云解决方案中，商业应用的关键功能和数据比之前更加公开，攻击面扩大了。

大多数用户需要的唯一客户端软件就是浏览器的时代就要到了（JS在吞噬一切？）。各种不同的应用都依赖于一组共享的协议和技术，从而产生一组公共安全漏洞。

Web 应用的好处

几种技术因素和商业动机共同推动了互联网的革命：

  • HTTP
  • 每个用户的电脑和手机上都有浏览器
  • 如今的浏览器功能强大，可以构建丰富的用户界面
  • 用于开发 Web 应用程序的核心技术和语言足够简单

Web 应用安全

“网站是安全的”

作者通过数据说明，网站不安全。

安全核心问题：用户可以提交任意输入

因为客户端不受控制，因此，用户可以提交任意输入到 Web 应用的服务端。应用必须假设用户的输入是恶意的，从而必须采取措施确保骇客不能使用精心构造的数据来干扰应用程序的逻辑和行为，从而获取会应用的未授权访问。

这一核心问题体现在以下几个方面：

  1. 用户可以干扰客户端和服务端信息传输中的任意部分，包括，参数，HTTP头，cookies 等。在客户端采取的任何安全措施都可以轻松规避。
  2. 用户可以按照任意顺序发送请求，并且可以在应用程序的任意阶段提交参数，或者根本不提交。程序员对用户交互行为的任意假设都可能是错的。
  3. 用户不仅可以使用浏览器访问应用。还有很多其他工具可以和应用进行快速交互，产生大量请求来寻找漏洞。

大多数针对 Web 应用的攻击都涉及将精心设计的输入发送到服务器，从而导致应用出现设计者所未预料的行为。有一些例子如下：

  • 改变产品价格从而低价购买商品
  • 修改会话令牌来劫持其他经过认证的用户的会话
  • 删除特定参数，导致应用出现逻辑缺陷
  • 更改由后端数据库处理的输入，插入恶意数据查询访问隐私数据

新的安全边界

传统的基于 IP 的安全边界不适用了，因为 Web 服务是面对公开的。攻击者并不需要控制基础设施，只需要影响应用执行就足够了。所以，从服务端来看，安全从网络边界向后移动了，要保证 Web 应用的安全，就得保证 Web 应用的代码质量。

从另一方面看，当用户访问有缺陷的应用时，客户端安全也面临威胁。骇客可以控制服务端向客户端发送伪造信息。

Web 安全的未来

SQL 注入漏洞慢慢销声匿迹，新的更加微妙的漏洞依然可以发现。

总结

Web 安全的两个根本因素：

  • Web 应用的功能越来越多，领域越来越广，应用越来越复杂
  • 用户输入不受限制
